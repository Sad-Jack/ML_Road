# 45. Data Leakage: опасная ошибка в ML

## Зачем это нужно

Data Leakage — одна из самых опасных ошибок в машинном обучении.

Модель может показывать отличное качество на обучении и тестах, но плохо работать в реальности.

Почему так происходит?

Потому что в признаки случайно попадает информация, которой у модели **не должно было быть в момент предсказания**.

---

## Простая интуиция

Представь студента, готовящегося к экзамену. Честный вариант: изучает учебник, сдаёт экзамен по своим знаниям. Нечестный вариант: учебник плюс готовые ответы к конкретным билетам. Результат — отличный, но только в этот раз. На следующем экзамене с другими билетами — провал. В ML то же самое: модель получает подсказку из будущего или из самого ответа, кажется умной на val — но в production подсказки нет.

## Что это такое

**Data Leakage** — это ситуация, когда в обучение модели попадает информация, которая не должна быть доступна при реальном предсказании.

Проще:

> модель случайно получает подсказку к правильному ответу.

Из-за этого качество на validation/test может выглядеть слишком хорошим.

Но это качество нечестное.

---

## Как это работает

Допустим, мы хотим предсказать:

```text
купит пользователь товар или нет
```

Target:

```text
bought = 1 / 0
```

В данных есть колонка:

```text
purchase_date
```

Проблема: дата покупки появляется только после покупки.

Если использовать `purchase_date` как признак, модель почти напрямую узнает ответ.

```text
есть purchase_date -> скорее всего bought = 1
нет purchase_date -> скорее всего bought = 0
```

Это утечка данных.

---

### Основные виды Data Leakage

#### 1. Target попал в признаки

Самый прямой вариант.

Например, мы предсказываем `price`, но создаём признак:

```text
price_per_m2 = price / area_m2
```

`price` — это target.
Значит, модель получила часть правильного ответа внутри признака.

Так делать нельзя.

---

### 2. Temporal leakage: признак появился после события

**Temporal leakage** (временной leakage) — особый вид утечки: обучающие данные содержат информацию из будущего относительно момента предсказания (например, использование данных после целевого события для предсказания этого события). Иначе говоря, признак становится известен только после того, что мы предсказываем — модель обучается на данных, которых у неё не будет в момент реального предсказания.

Пример:

```text
bought
purchase_date
delivery_status
final_payment_amount
```

Если мы предсказываем покупку заранее, нельзя использовать данные, которые появились после покупки.

Главный вопрос:

> эта колонка была бы известна в момент предсказания?

Если нет — её нельзя использовать.

---

### 3. Обработка данных сделана до split

Иногда утечка появляется не из-за колонки, а из-за неправильного порядка действий.

Например, нельзя сначала посчитать среднее по всему датасету, заполнить пропуски, а потом делить данные на train/test.

Плохо:

```text
весь датасет -> fill missing -> split
```

Правильно:

```text
split -> fit обработки на train -> применить к validation/test
```

Иначе информация из test попадёт в train через обработку данных.

---

### 4. Дубликаты между train и test

Если одинаковые или почти одинаковые объекты попали и в train, и в test, модель может просто запомнить их.

Качество будет выглядеть хорошим, но проверка уже не будет честной.

---

## Простой пример

Задача:

```text
предсказать, уйдёт ли клиент из сервиса
```

Target:

```text
churn = 1 / 0
```

Есть признаки:

| Признак                   | Можно использовать? | Почему                                 |
| ------------------------- | ------------------- | -------------------------------------- |
| age                       | да                  | известен заранее                       |
| plan_type                 | да                  | известен заранее                       |
| orders_count_last_30_days | да                  | если считается до момента предсказания |
| cancellation_date         | нет                 | появляется после ухода                 |
| final_status              | нет                 | напрямую связан с ответом              |

`cancellation_date` и `final_status` дают модели подсказку из будущего. Это Data Leakage.

Что происходит в production: модель запрашивает `cancellation_date` для нового пользователя — поля ещё нет, он не ушёл. Модель видит пустое значение вместо привычного сигнала и возвращает случайный результат. F1 на val был 0.97 — в production упал до 0.52. Хуже монетки.

---

### Как находить Data Leakage

Перед тем как добавить признак в `X`, задай три вопроса:

1. Эта колонка была бы известна в момент предсказания?
2. Не содержит ли она target напрямую или косвенно?
3. Не появилась ли она после события, которое мы пытаемся предсказать?

Если есть сомнение — признак нужно проверить внимательнее.

---

### Почему это опасно

Data Leakage опасна тем, что создаёт иллюзию хорошей модели.

Модель может показывать:

```text
accuracy = 0.99
```

Но не потому, что она нашла настоящие закономерности.
А потому что получила нечестную подсказку.

В реальности качество резко упадёт, потому что таких подсказок уже не будет.

---

## Практика

1. Дата-сайентист показывает тебе модель одобрения кредита: val F1 = 0.97. Ты подозреваешь Data Leakage. Назови три конкретных вопроса — по одному на каждый вид leakage: target, temporal, preprocessing — которые ты задашь аналитику, чтобы выяснить, что именно пошло не так.
2. Аналитик строит модель предсказания зарплаты (`salary`). Среди признаков добавляет `salary_band` — категорию: 'low' если salary < 50k, 'high' если salary ≥ 50k. Val R² = 0.99. Объясни точный механизм leakage: почему `salary_band` передаёт модели ответ и чем это отличается от честного категориального признака.
3. Два аналитика обрабатывают 5 000 строк (800 пропусков в признаке `income`). Аналитик А считает медиану до split по всем строкам — 52 400 ₽. Аналитик Б делает split сначала и считает медиану только по 4 000 train строкам — 52 100 ₽. Коллега говорит: «300 рублей — не важно». Объясни, почему это не вопрос точности числа, а вопрос принципа — какие строки «попали» в медиану А и почему это делает val-метрику нечестной.
4. В уроке «Как выбирать признаки для X» мы разбирали, что признак должен быть известен до момента предсказания. Вы строите модель одобрения кредита. Среди признаков есть `loan_amount` (запрошенная клиентом сумма) и `approved_amount` (одобренная банком сумма). Какой из двух признаков является утечкой и почему?
5. В уроке «Baseline-модель» мы видели, что даже простейшая модель имеет определённый уровень качества. Простая модель неожиданно показала accuracy = 0.99 на val-выборке в задаче, где baseline около 0.70. Назовите два признака того, что такой результат — следствие Data Leakage, а не реального качества модели.

## Краткий итог

- Симптом leakage — нереалистично высокая метрика: 0.99 там, где 0.80 уже хорошо.
- Три вида: target leakage (ответ в признаках), temporal leakage (будущие данные), preprocessing leakage (статистика test в train).
- Главный вопрос для каждого признака: «Была бы эта информация известна в момент предсказания?»
- Лечение preprocessing leakage — всегда Pipeline: сначала split, потом fit обработки только на train.

## Ключевые термины урока

- **Data Leakage** — симптом: нереалистично высокая метрика на val (99.9% там, где 80% — уже хорошо). Коварство: модель кажется отличной, ломается только в production — там «будущих» данных нет.
- **Target leakage** — признак вычислен из target или становится известен только одновременно с ним. Главный вопрос: «Был бы этот признак известен в момент предсказания?» Если нет — это leakage.
- **Temporal leakage** — в задачах с временными рядами shuffle=True при split = leakage: train видит «будущие» примеры. Лечение: TimeSeriesSplit или разбивка по дате вручную.
- **Preprocessing leakage** — scaler/imputer обучен на всём датасете до split: test просочился в train через статистики. Лечение: всегда Pipeline + сначала split, потом fit.
