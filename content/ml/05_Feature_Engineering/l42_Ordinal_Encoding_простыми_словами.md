# 42. Ordinal Encoding простыми словами

## Зачем это нужно

Из урока «Label Encoding» мы знаем, что Label Encoding опасен ложным порядком для неупорядоченных категорий. **Ordinal Encoding** — правильный способ для категорий, где порядок действительно есть: `small < medium < large`, `junior < senior`. Числа здесь несут реальный смысл.

---

## Простая интуиция

Представьте шкалу остроты блюда в меню ресторана: мягкое, среднее, острое, очень острое. Числа здесь несут настоящий смысл — «очень острое» больше «острого», «острое» больше «среднего». Закодировать 1/2/3/4 — совершенно логично. Но взять то же кодирование для названий блюд (борщ=1, паста=2, суши=3) — уже бессмыслица: борщ не «меньше» пасты.

## Что это такое

**Ordinal Encoding** — это способ кодирования категориальных признаков, где категории заменяются числами с учётом их порядка.

Пример:

| size   | size_encoded |
| ------ | -----------: |
| small  |            0 |
| medium |            1 |
| large  |            2 |

Так модель получает числовой признак, где большее число действительно означает "выше", "больше" или "сильнее" по смыслу.

---

## Как это работает

### 1. Берём признак с порядком

Например:

```text
education_level = school / bachelor / master / phd
```

Здесь порядок есть:

```text
school < bachelor < master < phd
```

---

### 2. Задаём правильную последовательность

Важно не доверять случайному порядку из данных.

Нужно явно сказать:

```text
school   -> 0
bachelor -> 1
master   -> 2
phd      -> 3
```

---

### 3. Заменяем категории числами

Было:

| education_level |
| --------------- |
| bachelor        |
| phd             |
| school          |
| master          |

Стало:

| education_level_encoded |
| ----------------------: |
|                       1 |
|                       3 |
|                       0 |
|                       2 |

Теперь модель может использовать этот признак как число.

---

## Простой пример

Есть признак качества товара:

```text
quality = low / medium / high
```

У него есть порядок:

```text
low < medium < high
```

Можно закодировать так:

| quality | quality_encoded |
| ------- | --------------: |
| low     |               0 |
| medium  |               1 |
| high    |               2 |

Это логично, потому что `high` действительно выше, чем `medium`, а `medium` выше, чем `low`.

**Ключевой вывод:** если применить вместо Ordinal случайный Label Encoding (`good=0, high=1, low=2`), линейная модель получит перевёрнутый или бессмысленный порядок и будет делать хуже предсказания для товаров с признаком `quality`. Правильный маппинг — не деталь, а часть смысла признака.

---

### Ordinal Encoding vs Label Encoding

Оба способа заменяют категории числами, но смысл разный.

| Способ           | Что делает                              | Главный риск                   |
| ---------------- | --------------------------------------- | ------------------------------ |
| Label Encoding   | даёт категориям числовые коды           | может создать ложный порядок   |
| Ordinal Encoding | кодирует категории по реальному порядку | нужно правильно задать порядок |

Пример с городами:

```text
Berlin -> 0
Hamburg -> 1
Munich -> 2
```

Для городов это плохо, потому что порядка нет.

А вот для уровня:

```text
junior -> 0
middle -> 1
senior -> 2
```

это нормально, потому что порядок есть.

---

### Когда Ordinal Encoding подходит

Ordinal Encoding хорошо подходит для признаков вроде:

| Признак      | Порядок                            |
| ------------ | ---------------------------------- |
| size         | small < medium < large             |
| level        | beginner < intermediate < advanced |
| education    | school < bachelor < master < phd   |
| satisfaction | low < medium < high                |
| priority     | low < normal < high < critical     |

Главное условие:

> категории должны иметь понятный естественный порядок.

Если порядка нет, лучше использовать One-Hot Encoding.

---

### Важный момент

Ordinal Encoding сохраняет порядок, но не всегда точно передаёт расстояние между категориями.

Например:

```text
junior = 0
middle = 1
senior = 2
```

Порядок правильный.

Но это не значит, что разница между `junior` и `middle` точно такая же, как между `middle` и `senior`.

Числа здесь показывают порядок, а не точное расстояние.

---

## Практика

1. Коллега кодирует `satisfaction_level = bad / ok / good / excellent` через LabelEncoder и получает алфавитный порядок: bad=0, excellent=1, good=2, ok=3. Что здесь сломалось? Как исправить через OrdinalEncoder?
2. Коллега применил OrdinalEncoder к признаку `region = North / South / East / West` и задал алфавитный порядок: East=0, North=1, South=2, West=3. После обучения val accuracy = 0.79; после замены на One-Hot Encoding — 0.84. Объясни, что именно OrdinalEncoder сделал неправильно с точки зрения смысла данных, и почему One-Hot сработал лучше для этого признака.
3. Вы применяете OrdinalEncoder к признаку `priority` и ошибочно задаёте порядок `['high', 'normal', 'low']` вместо обратного. Как это скажется на качестве модели и как такую ошибку можно обнаружить?
4. В уроке «Линейная регрессия» мы видели, что она воспринимает числовые признаки как расстояния. В датасете есть признак `clothing_size = XS / S / M / L / XL / XXL`. Предложите правильный маппинг. Что нужно проверить перед тем, как использовать этот признак в LinearRegression после кодирования?
5. В уроке «Label Encoding» мы разбирали, что числа создают иллюзию равных расстояний между категориями. Признак `satisfaction = low / medium / high`. Вы закодировали: low=1, medium=2, high=3. Можно ли утверждать, что разница между low и medium такая же, как между medium и high? Что это говорит об ограничениях Ordinal Encoding?

## Краткий итог

* Ordinal Encoding превращает категории в числа с учётом порядка.
* Он подходит только для признаков, где порядок реально есть.
* Примеры: размер, уровень, приоритет, образование, качество.
* Для категорий без порядка лучше использовать One-Hot Encoding.
* Числа в Ordinal Encoding показывают порядок, но не всегда точное расстояние между категориями.

## Ключевые термины урока

- **Ordinal Encoding** — как Label Encoding, но **порядок задаётся явно**. Разница принципиальная: Label Encoding присваивает числа произвольно, Ordinal — по заданному вами правилу.
- **Порядок категорий** — должен быть задан **вручную**, не доверяя алфавитному порядку из данных. Ошибка в маппинге тихо искажает сигнал: `high=1, low=2` перевернёт порядок и даст обратный смысл.
- **OrdinalEncoder (sklearn)** — принимает параметр `categories=[['low', 'medium', 'high']]`, фиксируя нужный порядок. В отличие от `LabelEncoder`, работает с 2D-массивом признаков.
- **Маппинг** — явное соответствие категория → число. Хранить его стоит отдельно (словарём или в `OrdinalEncoder`), чтобы воспроизвести в production без ошибок.
