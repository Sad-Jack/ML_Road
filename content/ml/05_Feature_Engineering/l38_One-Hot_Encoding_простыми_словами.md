# 38. One-Hot Encoding простыми словами

## Зачем это нужно

В уроке 37 мы разобрали, что категориальные признаки нужно кодировать. **One-Hot Encoding** — один из основных способов: подходит для категорий без естественного порядка и не создаёт ложной шкалы.

---

## Простая интуиция

Допустим, есть признак:

```text
city = Berlin
```

Модель не понимает строку `Berlin` как число.

Можно было бы сделать так:

```text
Berlin = 1
Hamburg = 2
Munich = 3
```

Но это плохая идея для городов, потому что у них нет порядка.
`Munich` не "больше" `Hamburg`, а `Hamburg` не "больше" `Berlin`.

One-Hot Encoding решает это иначе:

```text
city_Berlin = 1
city_Hamburg = 0
city_Munich = 0
```

То есть мы создаём отдельный признак для каждой категории.

---

## Что это такое

**One-Hot Encoding** — это способ кодирования категорий, где каждая категория превращается в отдельную колонку.

Если объект относится к категории — ставим `1`.
Если не относится — ставим `0`.

Было:

| city    |
| ------- |
| Berlin  |
| Hamburg |
| Munich  |
| Berlin  |

Стало:

| city_Berlin | city_Hamburg | city_Munich |
| ----------: | -----------: | ----------: |
|           1 |            0 |           0 |
|           0 |            1 |           0 |
|           0 |            0 |           1 |
|           1 |            0 |           0 |

---

## Как это работает

### 1. Берём категориальный признак

Например:

```text
device = mobile / desktop / tablet
```

---

### 2. Создаём отдельную колонку для каждой категории

Получаем:

```text
device_mobile
device_desktop
device_tablet
```

---

### 3. Заполняем колонки нулями и единицами

Если у объекта `device = mobile`, то:

```text
device_mobile = 1
device_desktop = 0
device_tablet = 0
```

Если `device = desktop`, то:

```text
device_mobile = 0
device_desktop = 1
device_tablet = 0
```

---

## Простой пример

Есть данные:

| user_id | device  | bought |
| ------: | ------- | -----: |
|       1 | mobile  |      1 |
|       2 | desktop |      0 |
|       3 | mobile  |      1 |
|       4 | tablet  |      0 |

`device` — категориальный признак.
`bought` — target.

После One-Hot Encoding:

| user_id | device_mobile | device_desktop | device_tablet | bought |
| ------: | ------------: | -------------: | ------------: | -----: |
|       1 |             1 |              0 |             0 |      1 |
|       2 |             0 |              1 |             0 |      0 |
|       3 |             1 |              0 |             0 |      1 |
|       4 |             0 |              0 |             1 |      0 |

Теперь модель получает числа и может использовать `device` как признак.

---

### Пример в Pandas

```python
import pandas as pd

df = pd.DataFrame({
    "device": ["mobile", "desktop", "mobile", "tablet"],
    "bought": [1, 0, 1, 0]
})

df_encoded = pd.get_dummies(df, columns=["device"])
print(df_encoded)
```

Результат будет примерно таким:

```text
bought  device_desktop  device_mobile  device_tablet
1       0               1              0
0       1               0              0
1       0               1              0
0       0               0              1
```

---

### Когда One-Hot Encoding подходит

One-Hot Encoding хорошо подходит, если:

* категорий не слишком много;
* между категориями нет естественного порядка;
* модель не умеет работать со строками напрямую;
* нужно сохранить смысл "это просто разные варианты".

Примеры хороших признаков:

| Признак        | Категории                 |
| -------------- | ------------------------- |
| device         | mobile / desktop / tablet |
| payment_method | card / cash / transfer    |
| color          | red / blue / green        |
| city           | Berlin / Hamburg / Munich |

---

### Когда One-Hot Encoding может быть неудобен

Если категорий очень много, One-Hot Encoding создаст слишком много новых колонок.

Например:

```text
city = 10 000 разных городов
```

После кодирования может появиться почти `10 000` новых признаков.

Это может:

* раздуть таблицу;
* замедлить обучение;
* увеличить расход памяти;
* сделать данные сложнее для обработки.

В таких случаях могут понадобиться другие способы кодирования.

---

### Важный момент про новые категории

Модель может столкнуться с категорией, которой не было на обучении.

Например, на train были:

```text
mobile
desktop
```

А в новых данных пришло:

```text
tablet
```

Если это не обработать заранее, могут возникнуть проблемы.

Обычно для таких случаев используют настройки encoder'а или отдельную категорию:

```text
unknown
```

Смысл: если категория неизвестна, не ломаем обработку, а относим её к "другим".

## Краткий итог

* One-Hot Encoding надёжен для категорий без порядка, но при высокой кардинальности создаёт curse of dimensionality.
* В колонках используются значения `0` и `1`.
* Метод хорошо подходит для категорий без естественного порядка.
* Он не создаёт ложную шкалу между категориями.
* При большом количестве категорий может появиться слишком много новых признаков.
* Новые неизвестные категории нужно учитывать заранее.

## Ключевые термины урока

- **One-Hot Encoding** — метод кодирования, создающий по одной бинарной колонке на каждую категорию.
- **Бинарные колонки** — колонки со значениями 0 или 1, обозначающие наличие категории.
- **OneHotEncoder (sklearn)** — класс для применения One-Hot Encoding в Pipeline.
- **get_dummies (pandas)** — функция pandas для быстрого One-Hot кодирования.
