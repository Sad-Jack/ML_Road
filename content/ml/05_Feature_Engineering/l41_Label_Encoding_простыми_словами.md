# 41. Label Encoding простыми словами

## Зачем это нужно

Из урока «Кодирование категориальных признаков» мы знаем, что категориальные признаки нужно кодировать. **Label Encoding** — самый простой способ: каждой категории присваивается число. Простой, но с важным риском ложного порядка.

---

## Простая интуиция

Представьте номерки в гардеробе: 1, 2, 3... Пальто под номером 100 не хуже и не лучше пальто под номером 1 — это просто метки для идентификации. Но если кто-то решит, что «чем больше номер, тем дороже пальто» — это бессмыслица. Именно такую ошибку делает линейная модель, когда видит Label Encoding для городов: Munich=2 воспринимается как «больше», чем Berlin=0.

## Что это такое

**Label Encoding** — это способ кодирования категорий, при котором каждому уникальному значению присваивается отдельное число.

Пример:

| city    | city_encoded |
| ------- | -----------: |
| Berlin  |            0 |
| Hamburg |            1 |
| Munich  |            2 |
| Berlin  |            0 |

Модель уже получает не строки, а числа.

Но здесь есть важный нюанс: эти числа не всегда имеют смысл как порядок.

---

## Как это работает

### 1. Берём категориальный признак

---

### 2. Находим все уникальные категории

```text
Berlin
Hamburg
Munich
```

---

### 3. Каждой категории даём номер

```text
Berlin  -> 0
Hamburg -> 1
Munich  -> 2
```

---

### 4. Заменяем строки на числа

Было:

| city    |
| ------- |
| Berlin  |
| Munich  |
| Hamburg |
| Berlin  |

Стало:

| city_encoded |
| -----------: |
|            0 |
|            2 |
|            1 |
|            0 |

---

### Важный момент

Label Encoding может создать ложный порядок.

Например:

```text
Berlin = 0
Hamburg = 1
Munich = 2
```

Модель может воспринять это так, будто:

```text
Munich > Hamburg > Berlin
```

Но у городов нет такого смысла.
Город с кодом `2` не "больше" города с кодом `1`.

Поэтому Label Encoding не всегда подходит для обычных неупорядоченных категорий.

---

### Когда Label Encoding подходит

Label Encoding можно использовать, когда категории имеют естественный порядок.

Например:

```text
low < medium < high
```

Или:

```text
junior < middle < senior
```

Тогда числа отражают реальный смысл:

| level  | encoded |
| ------ | ------: |
| junior |       0 |
| middle |       1 |
| senior |       2 |

Здесь порядок важен и понятен.

---

### Label Encoding vs One-Hot Encoding

| Способ           | Что делает                                     | Когда подходит                                                         |
| ---------------- | ---------------------------------------------- | ---------------------------------------------------------------------- |
| Label Encoding   | Заменяет категорию одним числом                | Когда есть порядок или модель умеет нормально работать с такими кодами |
| One-Hot Encoding | Создаёт отдельную колонку для каждой категории | Когда порядка между категориями нет                                    |

Пример для `city`:

### Label Encoding

| city    | encoded |
| ------- | ------: |
| Berlin  |       0 |
| Hamburg |       1 |
| Munich  |       2 |

Может появиться ложный порядок.

### One-Hot Encoding

| city_Berlin | city_Hamburg | city_Munich |
| ----------: | -----------: | ----------: |
|           1 |            0 |           0 |
|           0 |            1 |           0 |
|           0 |            0 |           1 |

Порядок не создаётся.

---

## Простой пример

LabelEncoder присваивает коды алфавитно: Berlin=0, Hamburg=1, Munich=2. Случайный порядок — не отражение реальности.

Если теперь обучить LinearRegression и получить коэффициент для `city_encoded` = 0.3, это значит: «с каждым следующим городом по алфавиту вероятность покупки растёт на 0.3». Это бессмыслица — модель нашла артефакт кодирования, а не реальный сигнал.

---

### Важное уточнение

В `sklearn` `LabelEncoder` чаще используют для кодирования `y`, то есть target.

Например:

```text
cat -> 0
dog -> 1
bird -> 2
```

Для признаков `X` используют другие инструменты — с явным заданием порядка или с созданием бинарных колонок. Смысл похожий, но важно не путать, где именно применяется кодирование.

## Практика

1. LabelEncoder присвоил: Berlin=0, Hamburg=1, Munich=2 (алфавитный порядок). Через неделю в датасете появился новый город — Amsterdam. Как encoder, обученный только на train, обработает его при вызове `transform`? Что нужно сделать, чтобы модель не сломалась?
2. Команда обучила Random Forest и линейную регрессию на признаке `device_type` с Label Encoding: mobile=0, tablet=1, desktop=2. Random Forest — val accuracy 0.81, линейная регрессия — 0.64. Коллега объяснил разрыв «сложностью задачи». Предложи альтернативное объяснение, связанное именно с кодированием, — и объясни, почему Random Forest менее чувствителен к Label Encoding неупорядоченных категорий, чем линейная модель.
3. Random Forest и LogisticRegression по-разному реагируют на Label Encoding неупорядоченных категорий. Для какой из двух моделей это менее опасно и почему?
4. В уроке «One-Hot Encoding» мы видели, что этот способ создаёт отдельную колонку для каждой категории. В чём преимущество Label Encoding перед One-Hot Encoding для признака с 1000 уникальных значений?
5. В уроке «Кодирование категориальных признаков» мы разбирали, что упорядоченные категории можно кодировать числами. Признак `priority = low / normal / high / critical` имеет реальный порядок. Объясните, почему здесь Label Encoding с правильным маппингом разумнее, чем создание бинарных колонок для каждого уровня.

## Краткий итог

- Используйте Label Encoding только когда порядок реален: `low < medium < high`. Для городов, цветов, типов устройств — опасно.
- Для деревьев и бустинга Label Encoding неупорядоченных категорий менее критичен: они разбивают по порогам, а не вычисляют расстояния.
- Для линейных моделей Label Encoding неупорядоченных категорий создаёт ложные коэффициенты — лучше One-Hot Encoding.
- `LabelEncoder` из sklearn — для `y` (target), не для признаков. Для признаков с порядком используйте инструменты, которые позволяют явно задать нужную последовательность категорий.

## Ключевые термины урока

- **Label Encoding** — один числовой признак вместо N бинарных колонок. Безопасен только если: (1) признак действительно имеет порядок (ordinal), или (2) используются деревья, которые нечувствительны к численному масштабу.
- **Числовой код** — порядок присвоения кодов случаен (алфавитный по умолчанию). Berlin=0, Hamburg=1, Munich=2 — это не «рейтинг городов», хотя линейная модель интерпретирует именно так. Для признаков с >50 уникальными значениями без порядка — рассмотрите target encoding (способ за рамками этого курса) вместо label encoding.
- **Ложный порядок** — главная ловушка. Линейные модели и нейросети интерпретируют числа как расстояния: код «2» воспринимается как «вдвое больше, чем код 1». Деревья менее чувствительны, но тоже могут строить разбиения на основе числового значения, а не смысла.
- **LabelEncoder (sklearn)** — предназначен для кодирования `y` (target), а не признаков. Для признаков используйте `OrdinalEncoder` с явным `categories=[...]`: он гарантирует, что порядок кодов совпадёт с реальным порядком категорий, а не будет случайно алфавитным.
