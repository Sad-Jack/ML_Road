# 36. Пропуски в данных

## Зачем это нужно

В реальных данных часто чего-то не хватает:

* человек не указал возраст;
* у товара нет рейтинга;
* у квартиры не заполнен этаж;
* у пользователя нет истории покупок;
* часть данных потерялась при сборе.

Для модели это проблема: многие алгоритмы не умеют работать с пустыми значениями напрямую.
Поэтому перед обучением нужно решить, что делать с пропусками.

---

## Простая интуиция

Представьте больничную карту, где часть граф не заполнена: у одного пациента нет записи об аллергиях, у другого — результатов анализов. Пустая клетка — не то же самое, что «нет аллергий» или «анализы в норме». Это просто «неизвестно». И прежде чем что-то заполнить в пустую клетку, нужно понять: данных нет потому что их не собирали, или потому что их действительно нет?

## Что это такое

**Пропуск в данных** — это отсутствующее значение в таблице.

В Python и Pandas такие значения часто отображаются как:

```text
NaN
None
null
```

Пример:

| age | city    | bought |
| --: | ------- | -----: |
|  25 | Berlin  |      1 |
| NaN | Hamburg |      0 |
|  31 | NaN     |      1 |

Здесь пропущены:

* `age` во второй строке;
* `city` в третьей строке.

---

## Как это работает

Пропуски могут появиться по разным причинам:

| Причина                  | Пример                                  |
| ------------------------ | --------------------------------------- |
| Человек не заполнил поле | не указал возраст                       |
| Данных ещё нет           | пользователь не сделал ни одной покупки |
| Ошибка сбора данных      | значение не записалось                  |
| Поле не подходит объекту | у нового товара ещё нет рейтинга        |

Важно понимать причину, потому что от неё зависит способ обработки.

---

### Как работать с пропусками

#### 1. Удалить строки с пропусками

Самый простой вариант — удалить объекты, где чего-то не хватает.

```text
если строка содержит NaN -> удалить строку
```

Это подходит, если:

* пропусков мало;
* удаление не испортит датасет;
* строки с пропусками не слишком важны.

Минус: можно потерять полезные данные.

---

### 2. Заполнить числовые пропуски

Если пропуск в числовой колонке, его можно заменить:

* средним значением;
* медианой;
* нулём;
* специальным значением.

Пример:

| age |
| --: |
|  25 |
| NaN |
|  31 |

Можно заменить `NaN` средним возрастом:

| age |
| --: |
|  25 |
|  28 |
|  31 |

Часто для чисел используют **медиану**, потому что она меньше зависит от выбросов.

---

### 3. Заполнить категориальные пропуски

Если пропуск в категории, можно заменить его:

* самым частым значением;
* отдельной категорией `unknown`.

Пример:

| city    |
| ------- |
| Berlin  |
| NaN     |
| Hamburg |

После обработки:

| city    |
| ------- |
| Berlin  |
| unknown |
| Hamburg |

Так модель видит, что город не был указан.

---

### 4. Добавить признак "был ли пропуск"

Иногда сам факт пропуска несёт полезный сигнал.

Например, если пользователь не указал доход, это может быть связано с его поведением.

Можно добавить отдельную колонку:

| income | income_was_missing |
| -----: | -----------------: |
|  50000 |                  0 |
|      0 |                  1 |
|  70000 |                  0 |

`income_was_missing = 1` показывает, что исходное значение было пропущено.

---

### Важный порядок обработки

Если мы заполняем пропуски средним, медианой или самым частым значением, это преобразование нужно считать **только на train**.

Правильно:

```text
train -> посчитали значение для заполнения
validation/test -> применили это же значение
```

Нельзя сначала заполнить пропуски на всём датасете, а потом делать split.
Иначе информация из validation/test может попасть в train.

---

## Простой пример

Есть колонка `orders_count` — количество заказов клиента. Часть значений — NaN. Медиана по остальным значениям = 4.

Аналитик заполняет NaN медианой, не задумываясь. Итог: новому пользователю, который ещё ничего не заказывал, приписали 4 заказа. Модель теперь считает его «опытным покупателем».

Что было нужно сначала понять:

- Если NaN означает «заказов не было» (новый пользователь) → заполнять нулём.
- Если NaN означает «не записали» (технический сбой) → заполнять медианой.
- Если сам факт пропуска несёт сигнал → добавить признак `orders_count_was_missing`.

**Вывод:** один и тот же NaN в одной колонке может требовать разной обработки в зависимости от смысла. Сначала — понять причину пропуска, потом — выбирать метод.

## Практика

1. В датасете e-commerce 40% пользователей не имеют значения в колонке `last_purchase_date`. Возможны два объяснения: (а) данные не собирались, (б) пользователь никогда не покупал. Как разные объяснения влияют на выбор способа обработки пропуска?
2. Колонка `age` имеет 15% пропусков. Почему не стоит автоматически удалять все строки с пропусками в этой колонке?
3. Когда для числовой колонки лучше заполнить пропуск медианой, а не средним?
4. Scaler нужно обучать только на train-выборке. Почему то же правило применяется к заполнению пропусков: нельзя считать медиану по всему датасету до split?
5. В уроке «Как выбирать признаки для X» мы разбирали, что признак должен нести полезный смысл о поведении объекта. У пользователей в данных не указан доход (`income = NaN`). Вы считаете, что сам факт пропуска несёт смысл — например, говорит об уровне дохода. Как это использовать при подготовке признаков и почему простое заполнение медианой уничтожит этот сигнал?

## Краткий итог

* Пропуски — обычная часть реальных данных; их нужно обработать до обучения модели.
* Строки с пропусками можно удалять, но не всегда — сначала поймите причину пропуска.
* Числовые пропуски заполняют медианой (устойчива к выбросам) или средним; категориальные — `unknown`.
* Иногда полезно добавить отдельный признак, который показывает сам факт пропуска.
* Значения для заполнения нужно считать только на train, а не на всём датасете.

## Ключевые термины урока

- **Импутация** — замена NaN числом до обучения модели. Медиана устойчива к выбросам и точнее отражает типичного пользователя, чем среднее; для категорий — константа `unknown` лучше, чем мода (не создаёт ложного «большинства»).
- **Правило train-only** — imputer.fit() только на train; значение для заполнения берётся из train, не из всего датасета. Fit на всём датасете до split → статистика test просочилась в train.
- **Индикатор пропуска** — бинарный признак `has_X_nan` иногда несёт сигнал: «нет данных о доходе» — само по себе информация. Простое удаление или заполнение уничтожает этот сигнал.
- **Когда удалять строки** — если пропусков в признаке > 70–80%, признак малоинформативен; если строка пропущена почти целиком. Если пропусков < 5% — удаление строк допустимо без потери статистики.
