# 73. Мониторинг модели

## Зачем это нужно

Деплой модели — не финал, а начало. Модель в production работает с реальными данными, которые меняются. Без мониторинга вы не узнаете, что модель деградировала, пока не получите жалобы пользователей или бизнес-потери.

## Простая интуиция

Представьте пилота самолёта. Он не просто взлетает и спит до посадки — он постоянно смотрит на приборы. Температура двигателя, высота, скорость — все под контролем. Если что-то выходит за допустимые границы — сразу сигнал. Мониторинг ML — те же «приборы» для модели.

## Что это такое

- **Мониторинг производительности** — отслеживание метрик модели (accuracy, F1) с течением времени
- **Мониторинг данных** — отслеживание изменений в распределении входных данных
- **Feature drift** — изменение распределения отдельных признаков
- **Prediction drift** — изменение распределения предсказаний модели
- **Latency мониторинг** — время ответа API
- **Алертинг** — автоматические уведомления при превышении порогов

## Как это работает

```python
# 1. Мониторинг дрейфа признаков
def check_drift(train_col, prod_col, threshold=0.15):
    drift = abs(prod_col.mean() - train_col.mean()) / abs(train_col.mean())
    return drift > threshold

if check_drift(X_train["age"], X_prod["age"]):
    print("АЛЕРТ: дрейф признака 'age'!")

# 2. Мониторинг предсказаний
pred_rate = y_pred.mean()   # доля позитивных предсказаний
# Если pred_rate резко изменился → возможен concept drift

# 3. Мониторинг качества (если есть реальные метки)
from sklearn.metrics import accuracy_score
current_acc = accuracy_score(y_true, y_pred)
print(f"Accuracy за период: {current_acc:.3f}")
```

## Простой пример

Телеком-компания деплоит модель предсказания оттока. Настроены четыре дашборда: (1) accuracy на недельной выборке с метками — снизилась с 86% до 81% за 3 месяца; (2) распределение возраста клиентов в запросах — среднее выросло с 38 до 44 лет (feature drift); (3) доля предсказаний «уйдёт» — выросла с 12% до 19% (prediction drift); (4) latency API — стабильно 28 мс. Алерт сработал на признаке «возраст» в октябре — команда начала подготовку к переобучению на свежих данных.

## Краткий итог

- Деплой — не финал: без мониторинга модель деградирует незаметно, пока бизнес-метрики не покажут проблему
- Мониторьте: качество модели, распределение признаков, предсказания, latency
- Автоматизируй обнаружение drift: вручную отслеживать сотни признаков в production нереально
- Логируйте всё: входные данные, предсказания, метки — для переобучения

## Ключевые термины урока

- **Мониторинг модели** — непрерывная проверка качества, распределения данных и производительности после деплоя.
- **Feature drift** — изменение распределения входных признаков относительно обучения.
- **Prediction drift** — изменение распределения предсказаний модели.
- **KS-тест (Kolmogorov-Smirnov)** — статистический тест для обнаружения изменения распределения.
- **Алертинг** — автоматическое уведомление при обнаружении критического отклонения метрик.
