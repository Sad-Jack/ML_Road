# 75. Мониторинг модели

## Зачем это нужно

Деплой модели — не финал, а начало. Модель в production работает с реальными данными, которые меняются. Без мониторинга вы не узнаете, что модель деградировала, пока не получите жалобы пользователей или бизнес-потери.

## Простая интуиция

Представьте пилота самолёта. Он не просто взлетает и спит до посадки — он постоянно смотрит на приборы. Температура двигателя, высота, скорость — все под контролем. Если что-то выходит за допустимые границы — сразу сигнал. Мониторинг ML — те же «приборы» для модели.

## Что это такое

- **Мониторинг производительности** — отслеживание метрик модели (accuracy, F1) с течением времени
- **Мониторинг данных** — отслеживание изменений в распределении входных данных
- **Feature drift** — изменение распределения отдельных признаков
- **Prediction drift** — изменение распределения предсказаний модели
- **KS-тест (Kolmogorov-Smirnov)** — стандартный статистический тест для сравнения двух распределений. Если распределение признака в production значимо отличается от обучающего (p < 0.05) — это feature drift. Именно его имеют в виду, когда говорят «обнаружить дрейф автоматически»
- **Latency мониторинг** — время ответа API
- **Алертинг** — автоматические уведомления при превышении порогов

## Как это работает

Мониторинг строится на нескольких уровнях. Первый — метрики качества: если есть реальные метки (с задержкой), считаем accuracy, F1 или MAE на недельных срезах и сравниваем с базовым уровнем. Второй — мониторинг входных данных: сравниваем распределение каждого признака в production с обучающим распределением. Для этого используют KS-тест или сравнение средних и квантилей. Если отклонение превышает порог — алерт. Третий — prediction drift: если доля положительных предсказаний резко изменилась — возможен concept drift. Четвёртый — latency: время ответа API должно оставаться в норме. Всё это логируется и визуализируется на дашбордах, а алерты отправляются команде автоматически.

## Простой пример

Телеком-компания деплоит модель предсказания оттока. Настроены четыре дашборда: (1) accuracy на недельной выборке с метками — снизилась с 0.86 до 0.81 за 3 месяца; (2) распределение возраста клиентов в запросах — среднее выросло с 38 до 44 лет (feature drift); (3) доля предсказаний «уйдёт» — выросла с 12% до 19% (prediction drift); (4) latency API — стабильно 28 мс. Алерт сработал на признаке «возраст» в октябре — команда начала подготовку к переобучению на свежих данных.

2. Что такое feature drift и как его обнаружить автоматически с помощью статистического теста?
3. Вы задеплоили модель предсказания оттока клиентов. Реальные метки приходят с задержкой 30 дней. Как мониторить модель в этот период?
4. В уроке «Почему модель в проде ≠ модель на обучении» мы разбирали data drift и concept drift. Как мониторинг входных признаков помогает отличить один тип от другого?
5. Почему важно логировать не только входные признаки, но и предсказания модели, и как эти данные используются при следующем переобучении модели?

## Краткий итог

- Деплой — не финал: без мониторинга модель деградирует незаметно, пока бизнес-метрики не покажут проблему
- Мониторьте: качество модели, распределение признаков, предсказания, latency
- Автоматизируйте обнаружение drift: вручную отслеживать сотни признаков в production нереально
- Логируйте всё: входные данные, предсказания, метки — для переобучения

## Ключевые термины урока

- **Мониторинг модели** — непрерывная проверка качества, распределения данных и производительности после деплоя.
- **Feature drift** — изменение распределения входных признаков относительно обучения.
- **Prediction drift** — изменение распределения предсказаний модели.
- **KS-тест (Kolmogorov-Smirnov)** — тест для сравнения двух распределений; обнаруживает, что признак в production стал распределён иначе, чем на обучении.
- **Алертинг** — автоматическое уведомление при обнаружении критического отклонения метрик.
