# 70. От данных до предсказания

## Зачем это нужно

Модель — это не только обучение и предсказание. Между сырыми данными и финальным предсказанием — цепочка трансформаций. Если нарушить её порядок, получим неправильные предсказания или data leakage. Pipeline в sklearn решает эту проблему автоматически.

## Простая интуиция

Думайте о конвейере на заводе. Сырой материал проходит через последовательность станков: очистка → нарезка → сборка → проверка качества. Каждый станок делает своё, порядок важен. ML-пайплайн — тот же конвейер: данные → imputer → scaler → модель.

## Что это такое

- **Pipeline** — последовательность трансформаций + финальная модель
- **Трансформеры** — шаги, которые делают fit_transform (scaler, imputer, encoder)
- **Estimator** — финальный шаг (модель), делающий fit и predict
- Ключевое правило: трансформеры должны обучаться ТОЛЬКО на train данных
- **ColumnTransformer** — применяет разные трансформации к разным столбцам

## Как это работает

Pipeline соединяет все шаги preprocessing и финальную модель в единый объект. При вызове `fit()` каждый трансформер обучается только на train-данных и запоминает свои параметры (например, scaler запоминает среднее и стандартное отклонение). При вызове `predict()` к новым данным применяются только уже зафиксированные параметры — без повторного обучения. ColumnTransformer внутри Pipeline позволяет одновременно применять разные трансформации к числовым и категориальным признакам. Весь объект Pipeline можно сохранить и загрузить — preprocessing и модель всегда едины.

## Простой пример

Команда строит модель оценки кредитного риска: 10 числовых признаков и 3 категориальных (профессия, регион, тип кредита). Без Pipeline: scaler обучен на всём датасете (data leakage), OneHotEncoder применён отдельно, при деплое забыли передать imputer. Результат: модель в production выдаёт ошибки при NaN. С Pipeline: один объект (imputer → ColumnTransformer → LogisticRegression) обучается только на train, сохраняется в файл, загружается в API. Новый клиент с пропущенным полем обрабатывается корректно — imputer внутри pipeline заполняет пропуск медианой.

## Практика

1. Коллега говорит: «Я уже применил scaler отдельно, зачем ещё Pipeline?» Что именно пойдёт не так при кросс-валидации, если scaler обучен вне Pipeline?
2. Чем `ColumnTransformer` отличается от обычного Pipeline и в каких случаях он необходим?
3. Вы обучили Pipeline (imputer → scaler → RandomForest). Коллега применил `scaler.fit_transform(X_test)` отдельно перед предсказанием. Что пошло не так?
4. В уроке «Data Leakage» мы разбирали, как информация из test попадает в обучение. Как Pipeline автоматически предотвращает самый частый вид leakage — когда scaler или imputer обучается на всём датасете, включая test?
5. В уроке «Overfitting и Underfitting» мы видели, что модель может «запомнить» train вместо обобщения. Почему правило «трансформеры обучаются только на train» — это тот же принцип, только применённый к шагам preprocessing?

## Краткий итог

- Pipeline соединяет preprocessing и модель в единый объект
- Трансформеры обучаются только на train, к test применяется только transform
- ColumnTransformer внутри Pipeline гарантирует: preprocessing обучается только на train и автоматически применяется к test
- Обученный Pipeline можно сохранить и использовать в production

## Ключевые термины урока

- **Pipeline** — единственный правильный способ применить scaler при кросс-валидации: fit() только на fold-train, а не на всём датасете. Без Pipeline scaler почти всегда обучается с data leakage.
- **Трансформер** — шаг Pipeline с методами fit_transform (на train) и transform (на test). Ключевое отличие от Estimator: трансформер не делает предсказаний.
- **Estimator** — финальный шаг Pipeline с методами fit и predict. В Pipeline может быть только один Estimator и только последним шагом.
- **ColumnTransformer** — позволяет применять разные трансформации к разным колонкам одновременно. Без него числовые и категориальные признаки пришлось бы обрабатывать отдельно, что нарушает целостность Pipeline.
