# 78. Как тестировать ML-системы

## Зачем это нужно

ML-системы — это не только ML-код: это данные, preprocessing, бизнес-логика, API. Традиционное тестирование ПО недостаточно. ML-системы требуют особых подходов к тестированию, учитывающих вероятностную природу моделей и зависимость от данных.

## Простая интуиция

Обычный тест проверяет: «при входе A всегда получаем B». В ML всё иначе: «при входе A получаем B примерно в 90% случаев». Тестировать нужно не только код, но и поведение модели: не пропускает ли она определённый тип объектов? Не слишком ли уверена в очевидных случаях?

## Что это такое

- **Unit tests** — тестирование отдельных функций preprocessing, feature engineering
- **Integration tests** — тестирование полного pipeline от сырых данных до предсказания
- **Data validation tests** — проверка схемы, типов, диапазонов входных данных
- **Model behavior tests** — проверка специфических свойств модели (например, если доход вырос — score не должен упасть)
- **Regression tests** — модель не ухудшилась после изменений

## Как это работает

Unit tests проверяют корректность отдельных функций: возвращает ли scaler правильную размерность, корректно ли кодируются категориальные признаки. Data validation tests проверяют каждый входящий запрос: есть ли нужные колонки, правильные ли типы, не выходят ли значения за допустимые диапазоны. При провале — запрос отклоняется до попадания в модель. Model behavior tests проверяют логические свойства предсказаний: рост дохода не должен повышать вероятность дефолта, изменение нерелевантного признака не должно менять предсказание. Regression tests запускаются перед каждым деплоем: новая версия модели должна показывать метрику не хуже предыдущей на hold-out выборке. Если тест провалился — деплой блокируется.

## Простой пример

Команда деплоит новую версию модели предсказания оттока в CI/CD пайплайне. Unit test: scaler не изменяет размерность (10 признаков на входе → 10 на выходе). Data validation: запрос с отрицательным возрастом или строкой вместо числа — возвращает 400 Bad Request, не падает с ошибкой. Model behavior test: для клиента с доходом 100 000 ₽ вероятность оттока не должна быть выше, чем для клиента с доходом 50 000 ₽ при одинаковых прочих признаках. Regression test: F1 на hold-out выборке не ниже 0.78. Всё прошло — деплой. Один тест провалился — пайплайн остановлен, деплой отменён.

## Практика

1. Коллега написал unit-тест, который проверяет, что predict() возвращает число без исключения. CI/CD проходит, но в production модель тихо выдаёт бессмысленные предсказания при NaN-входах. Какого теста не хватает и почему unit-тест этого не поймал?
2. Что проверяет data validation test и что должно произойти при его провале в production pipeline?
3. Вы деплоите модель кредитного скоринга. Придумайте: один unit test, один data validation test и один model behavior test для этой задачи.
4. В уроке «Мониторинг модели» мы разбирали, что мониторинг работает после деплоя. Как regression test дополняет мониторинг, автоматически блокируя деплой, если новая версия модели хуже предыдущей на ключевых метриках?
5. В уроке «Что такое модель в API» мы разбирали, что API принимает JSON и валидирует входные данные. Как data validation test при каждом запросе отличается от regression test по роли в ML-системе: что каждый из них защищает?

## Краткий итог

- ML-системы требуют: unit tests кода, data validation, model behavior tests, regression tests
- Data validation — проверяйте схему, типы и диапазоны входных данных
- Model behavior tests проверяют специфические свойства предсказаний
- Интегрируйте тесты в CI/CD: перед каждым деплоем все тесты должны проходить

## Ключевые термины урока

- **Unit tests для ML** — тестируют корректность кода, не качество модели: «feature engineering возвращает правильный тип?», «scaler применяется только на train?». Обычные pytest; не требуют обученной модели.
- **Data validation tests** — 4 проверки: схема (нужные колонки есть), типы (числа — числа, не строки), диапазоны (возраст 0–120), NaN в критичных полях. Ловят баги при изменении upstream данных — до того, как модель молча выдаст бред.
- **Model behavior tests (invariance tests)** — проверяют логику: изменение нерелевантного признака не меняет предсказание; более положительный сигнал → более высокий скор. Ловят логические баги, которые unit tests не поймают.
- **Regression tests** — новая модель ≥ старой на ключевых метриках перед деплоем; иначе деплой блокируется. CI/CD для ML: тест провалился = автоматический откат или запрет деплоя.
