# 69. Training vs Inference

## Зачем это нужно

Обучение модели и её использование для предсказаний — принципиально разные операции. В production нас интересует только inference: скорость, надёжность, стоимость предсказания. Путаница между training и inference — частая причина проблем при деплойменте.

## Простая интуиция

Повар учится готовить годами (training) — изучает рецепты, пробует разные ингредиенты, совершенствует технику. Но когда гость заказывает блюдо в ресторане, повар не учится заново — он просто готовит по уже выученному рецепту (inference). Ресторан не может ждать годы обучения для каждого заказа.

## Что это такое

- **Training** — процесс подбора параметров модели на обучающих данных. Медленно, ресурсоёмко, делается заранее
- **Inference** — использование обученной модели для предсказаний на новых данных. Должно быть быстрым
- **Batch inference** — предсказания для большого набора объектов сразу (ночная обработка)
- **Real-time inference** — предсказание для одного объекта с минимальной задержкой (API)
- **Latency** — задержка ответа при inference; критична для real-time систем
- **Throughput** — сколько предсказаний в секунду может обработать система

## Как это работает

```python
from sklearn.ensemble import RandomForestClassifier

# Training: один раз, медленно
model = RandomForestClassifier(n_estimators=100)
model.fit(X_train, y_train)          # секунды–минуты

# Batch inference: много объектов сразу
predictions = model.predict(X_test)  # миллисекунды

# Real-time inference: один объект за раз
pred = model.predict(X_test[0:1])    # единицы миллисекунд

# В production: train → сохранить (joblib) → загрузить → predict
```

## Простой пример

E-commerce платформа обучает модель предсказания вероятности покупки: Random Forest на 50 000 пользователей, обучение занимает 8 минут — это training. Деплой модели в real-time API: при загрузке страницы товара за 80 мс нужно выдать персонализированную цену. Random Forest предсказывает за 12 мс (один объект) — вписывается в бюджет. Batch inference: ночью для 200 000 пользователей предварительно считают рекомендации — throughput 3 000 предсказаний/сек, весь прогон занимает 67 секунд.

## Краткий итог

- Разделение обучения и inference — ключевое архитектурное решение: обучать заново на каждый запрос в production невозможно
- Latency inference определяет применимость модели: медленная модель убивает UX даже при отличном качестве предсказаний
- Latency (задержка) и throughput (пропускная способность) критичны в production
- Выбор алгоритма для production часто определяется требованиями к скорости

## Ключевые термины урока

- **Training** — процесс подбора параметров модели на обучающих данных; выполняется заранее.
- **Inference** — применение обученной модели к новым данным; должно быть быстрым.
- **Batch inference** — обработка большого набора данных за один раз.
- **Real-time inference** — предсказание для одного объекта за миллисекунды.
- **Latency** — время ответа модели на один запрос.
