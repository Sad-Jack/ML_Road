# 91. Что такое RAG

## Зачем это нужно

LLM не знает о событиях после даты обучения, не знает ваши внутренние документы и галлюцинирует факты. RAG (Retrieval-Augmented Generation) решает все три проблемы: добавляет актуальные знания без переобучения.

## Простая интуиция

Представьте эксперта с доступом к библиотеке. Вы задаёте вопрос — эксперт быстро находит нужные страницы из книг, читает их и отвечает, опираясь на прочитанное, а не только на своё образование. RAG — это LLM + умный поиск по базе знаний.

## Что это такое

- **RAG** (Retrieval-Augmented Generation) — архитектура: Retrieval (поиск) + Generation (генерация)
- **Retrieval** — поиск релевантных документов по запросу пользователя
- **Vector store** — база данных эмбеддингов документов (Chroma, Pinecone, FAISS)
- **Chunking** — разбивка документов на части нужного размера. Выбор размера критичен: слишком маленький chunk (50 токенов) → теряется контекст; слишком большой (2000+) → в одном chunk много шума, retrieval теряет точность. Оптимум обычно 200–500 токенов с небольшим перекрытием
- **Embedding model** — модель для векторизации текста (text-embedding-3-small и др.)
- **Контекст** — найденные документы передаются в LLM как часть промпта

## Как это работает

```python
# Схема RAG системы (концептуально)

# ЭТАП 1: ИНДЕКСИРОВАНИЕ (делается один раз)
# 1. Загрузить документы
documents = [
    "Компания основана в 2010 году. Штаб-квартира в Москве.",
    "Продукты: CRM система, аналитическая платформа, мобильное приложение.",
    "Политика возврата: 14 дней с момента покупки без объяснения причин."
]

# 2. Разбить на chunks (обычно 200-500 токенов с перекрытием)
# 3. Создать embeddings для каждого chunk
# 4. Сохранить в vector store

# ЭТАП 2: ЗАПРОС (при каждом вопросе)
def rag_pipeline(question, documents, top_k=2):
    """
    Упрощённая RAG система
    """
    # Поиск: находим top_k наиболее релевантных документов
    # В реальности: cosine similarity между embedding вопроса и документов
    # Здесь: простой поиск по ключевым словам
    keywords = question.lower().split()
    scored_docs = []
    for doc in documents:
        score = sum(1 for kw in keywords if kw in doc.lower())
        scored_docs.append((score, doc))
    scored_docs.sort(reverse=True)
    context = "\n".join([doc for _, doc in scored_docs[:top_k]])

    # Генерация: передаём контекст + вопрос в LLM
    prompt = f"""Используй только следующий контекст для ответа.

Контекст:
{context}

Вопрос: {question}
Ответ:"""
    return prompt

q = "Какова политика возврата?"
result = rag_pipeline(q, documents)
print("Промпт для LLM:")
print(result)
```

## Простой пример

Нефтегазовая компания строит RAG-систему для поиска по 12 000 технических регламентов (PDF, общий объём 80 000 страниц). Индексирование: каждый документ разбивается на chunks по 400 токенов с перекрытием 50 токенов, создаются эмбеддинги через text-embedding-3-small, сохраняются в FAISS (12 млн векторов). Запрос: «Каков регламент технического обслуживания компрессора K-201?» — система находит 3 релевантных chunk из разных документов за 80 мс, передаёт их в Claude. Ответ содержит конкретные цифры из регламентов со ссылками на источники. Без RAG — галлюцинации с неверными интервалами обслуживания.

## Краткий итог

- RAG = поиск релевантных документов + передача их в контекст LLM
- Решает проблемы: устаревшие знания, внутренние документы, галлюцинации
- Vector store хранит embeddings документов для семантического поиска
- Качество RAG в значительной мере определяется качеством chunking и retrieval

## Ключевые термины урока

- **RAG** — LLM без RAG отвечает из своих весов (статичные знания до cutoff date); RAG даёт LLM доступ к актуальным или приватным документам в реальном времени без переобучения.
- **Chunking** — chunk_size 200–500 токенов = баланс между контекстом и точностью поиска. Слишком маленький chunk теряет контекст абзаца; слишком большой → в найденном chunk много нерелевантного шума.
- **Vector store** — хранит embedding каждого chunk; при запросе embedding запроса сравнивается с embeddings чанков через cosine similarity (ANN-поиск). FAISS, ChromaDB, Weaviate — популярные реализации.
- **Retrieval** — если retrieval нашёл не тот документ, LLM будет уверенно генерировать на основе неверного контекста. Качество RAG определяется на 80% качеством retrieval, а не генерацией.
- **RAG vs Fine-tuning** — RAG для актуальных/обновляемых знаний (база обновляется без переобучения); fine-tuning для изменения стиля/поведения модели. Для корпоративной базы знаний — RAG; для медицинского диалекта — fine-tuning.
