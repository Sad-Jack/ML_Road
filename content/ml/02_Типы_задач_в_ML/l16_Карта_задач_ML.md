# 16. Карта задач ML

## Зачем это нужно

Когда приходит реальная задача, первый вопрос не «какой алгоритм взять», а «какой тип ML-задачи здесь нужен». Неправильный выбор типа задачи — это потеря времени и неверные метрики. Этот урок — справочная карта, к которой стоит возвращаться.

## Простая интуиция

Представь, что ты врач. Перед тем как назначить лечение, ты ставишь диагноз. В ML то же самое: сначала определяешь тип задачи, потом выбираешь инструмент. Правильная постановка — половина решения.

## Что это такое

Карта задач ML — структурированный обзор всех типов задач с логикой выбора между ними.

**Справочная таблица: условия → тип задачи**

| Есть метки? | Что нужно предсказать / найти | Тип задачи | Тип метрики |
|---|---|---|---|
| Да | Число | Регрессия | метрики отклонения от числового значения |
| Да | Категорию (да/нет, A/B/C) | Классификация | метрики качества разделения классов |
| Да | Порядок объектов | Ранжирование | CTR (Click-Through Rate — доля пользователей, кликнувших на объект из числа тех, кто его увидел), метрики позиции |
| Да (косвенно — поведение) | Что понравится пользователю | Рекомендации | CTR, конверсия |
| Нет | Группы похожих объектов | Кластеризация | — |
| Нет / мало | Что выбивается из нормы | Детекция аномалий | метрики, учитывающие пропущенные аномалии |

## Как это работает

**Фреймворк выбора — два вопроса:**

```
Есть размеченные данные?
├── НЕТ → Что нужно сделать?
│         ├── Найти группы → Кластеризация
│         └── Найти выбросы → Детекция аномалий
└── ДА → Что предсказываем?
          ├── Число → Регрессия
          ├── Категорию → Классификация
          ├── Порядок → Ранжирование
          └── Что понравится → Рекомендации
```

**Разборы реальных задач**

*Задача 1: «Предсказать цену автомобиля по его характеристикам»*
→ Есть метки (исторические цены)? **Да**
→ Что предсказываем? **Число**
→ **Регрессия**

*Задача 2: «Определить, уйдёт ли сотрудник из компании»*
→ Есть метки (история — кто уволился)? **Да**
→ Что предсказываем? **Категорию (уйдёт / останется)**
→ **Классификация**

*Задача 3: «Разбить клиентов на сегменты, никаких меток нет»*
→ Есть метки? **Нет**
→ Что нужно? **Найти группы**
→ **Кластеризация**

*Задача 4: «Найти мошеннические транзакции, размеченных примеров почти нет»*
→ Есть метки? **Почти нет**
→ Что нужно? **Найти выбросы**
→ **Детекция аномалий**

## Одна задача — несколько формулировок

Это важный момент: одна бизнес-задача может быть поставлена по-разному в зависимости от данных.

**Пример: «Найти нечестных игроков в онлайн-игре»**
- Есть размеченные примеры чит-игроков → **Классификация** (честный / нечестный)
- Меток нет, но есть поведенческие данные → **Детекция аномалий** (поведение выбивается из нормы)

**Пример: «Предсказать оценку фильма»**
- Нужна точная оценка 7.3, 8.1... → **Регрессия**
- Нужно только «понравится / нейтрально / не понравится» → **Классификация**

**Пример: «Улучшить выдачу товаров в поиске»**
- Нужен правильный порядок → **Ранжирование**
- Хочется персонализировать под пользователя → **Рекомендации**

Выбор формулировки зависит от того, какие данные есть и что реально важно бизнесу.

## Практика

1. Пройди фреймворк для каждой задачи и назови тип: (а) предсказать температуру воздуха завтра; (б) определить, кликнет ли пользователь на баннер; (в) сгруппировать статьи по темам без разметки; (г) рекомендовать треки в плейлист.
2. Задача: «Предсказать оценку клиента от 1 до 5». Как её можно сформулировать как регрессию? Как классификацию? Что нужно уточнить у заказчика?
3. Компания хочет найти аномальные транзакции. У них нет ни одного размеченного мошенничества. Какой тип задачи и почему классификация здесь не подойдёт?
4. В уроке «Вероятности в классификации» мы разобрали, что классификатор выдаёт вероятность для каждого класса. Посмотри на колонку «Что нужно предсказать / найти» в таблице: в каких типах задач выход модели — число или вероятность, а в каких — нет? Что это означает для способа оценки результата?
5. Мини-кейс: стартап хочет «улучшить пользовательский опыт в приложении». Это слишком размыто. Задай три уточняющих вопроса, которые помогут определить тип ML-задачи.

## Краткий итог

- Определить правильный тип задачи — первый и важнейший шаг
- Два ключевых вопроса: есть ли метки и что именно нужно получить на выходе
- Есть метки + число → регрессия; + категория → классификация; + порядок → ранжирование
- Нет меток → кластеризация (группы) или детекция аномалий (выбросы)
- Одна бизнес-задача может быть сформулирована несколькими способами — выбор зависит от данных
- Тип задачи определяет алгоритм, метрики, данные и архитектуру системы

## Ключевые термины урока

- **Карта задач ML** — фреймворк для выбора типа задачи. На практике тип задачи нередко меняется при уточнении требований: заказчик просит «предсказать оценку» (регрессия), но после разговора выясняется, что нужен только порядок (ранжирование). Умение быстро переформулировать — ключевой навык.
- **Регрессия** — выход модели не ограничен снизу и сверху: она может предсказать отрицательную цену или значение больше 100%. Если диапазон важен — нужен постобработчик или специальная функция потерь.
- **Классификация** — помимо класса модель выдаёт вероятность каждого класса (predict_proba). Это важно: устанавливая порог не 0.5, а, например, 0.3, можно ловить больше положительных случаев ценой ложных срабатываний.
- **Кластеризация** — нет единственно «правильного» числа кластеров: это гиперпараметр. Число кластеров подбирают через метрики качества (silhouette score) или, чаще, через бизнес-логику.
- **Детекция аномалий** — если размеченных примеров аномалий достаточно, лучше взять классификацию: она точнее. Детекция аномалий — это выбор по необходимости, когда меток нет или их единицы.
- **Ранжирование** — в отличие от классификации, важен не только факт ошибки, но и позиция: поставить плохой объект на 1-е место — критично, пропустить 2-й элемент на 3-е — терпимо. Поэтому метрики ранжирования (NDCG, MRR) взвешивают позиции.
