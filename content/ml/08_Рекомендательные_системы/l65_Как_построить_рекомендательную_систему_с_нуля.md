# 65. Как построить рекомендательную систему с нуля

## Зачем это нужно

Ты знаешь коллаборативную фильтрацию, контентную, implicit/explicit feedback, гибриды, холодный старт и метрики. Но когда приходит реальная задача — с чего начинать? Этот урок — пошаговая схема рассуждений, которую можно применять к любой задаче рекомендаций.

## Простая интуиция

Хороший врач не назначает лечение до осмотра. Хороший специалист по ML не выбирает алгоритм до понимания данных и задачи. Сначала вопросы — потом решение.

## Что это такое

Построение рекомендательной системы — это не выбор алгоритма, а последовательность решений: от понимания задачи и данных до запуска и оценки. Алгоритм — последнее, что выбираешь.

## Как это работает

**Шаг 1. Понять задачу**

Что нужно рекомендовать? Кому? С какой целью?
- Продать товар → оптимизируем конверсию
- Удержать пользователя → оптимизируем время в приложении
- Помочь найти нужное → оптимизируем релевантность

**Шаг 2. Разобраться с данными**

| Вопрос | Что это значит для системы |
|---|---|
| Есть история взаимодействий? | Можно использовать collaborative filtering |
| Feedback explicit или implicit? | Влияет на качество и интерпретацию сигналов |
| Много новых пользователей? | Нужно решение холодного старта |
| Объекты хорошо описаны (теги, категории)? | Можно использовать content-based |

**Шаг 3. Выбрать подход**

| Ситуация | Подход |
|---|---|
| Много истории + мало новых пользователей | Collaborative filtering |
| Мало истории + богатые метаданные объектов | Content-based |
| Много новых пользователей | Гибрид: CF для старых, CB для новых |
| Только запустились | Популярное + content-based |
| Нужен максимум качества | Гибрид с несколькими сигналами |

**Шаг 4. Решить проблему холодного старта**

Он есть всегда. Нужен план:
- Для новых пользователей: онбординг-вопросы, популярное, demographic fallback
- Для новых объектов: content-based сразу, принудительный показ для сбора данных

**Шаг 5. Определить метрики успеха**

Что считаем хорошим результатом? CTR? Конверсия? Время в приложении?
Определяем до запуска — не после.

**Шаг 6. Запустить, оценить, итерировать**

Начать с простого: популярное или content-based. Это всегда лучше, чем «ничего». Потом усложнять, измеряя каждое изменение через A/B тест.

## Простой пример

Стартап строит рекомендации для платформы онлайн-курсов: 15 000 студентов, 600 курсов. Шаг 1: цель — удержание (студент завершает курс). Шаг 2: есть история просмотров (implicit), описания курсов (категория, уровень, теги), 20% новых студентов ежемесячно. Шаг 3: гибрид — CF для студентов с историей 3+ курсов, content-based для новых. Шаг 4: холодный старт — онбординг-анкета 3 вопроса. Шаг 5: метрика — процент завершённых курсов. Шаг 6: запуск content-based за 2 недели, CF через месяц. Завершаемость выросла с 31% до 48%.

## Практика

1. Тебе пришла задача: построить рекомендации для новостного сайта. С каких вопросов ты начнёшь?
2. Стартап только запустил маркетплейс: 500 пользователей, 2 000 товаров, истории почти нет. Какой подход выбрать прямо сейчас?
3. Через 6 месяцев у того же маркетплейса 50 000 активных пользователей и богатая история покупок. Что изменится в подходе и почему?
4. Почему «начать с простого» — хорошая стратегия, а не признак слабого специалиста?
5. Мини-кейс: музыкальный стриминг. Данные: история прослушиваний (implicit), жанры треков, время суток, устройство пользователя. Пройди все 6 шагов и опиши свой подход.

## Краткий итог

- Начинай с задачи и данных, не с алгоритма
- Определи: есть ли история, какой тип feedback, насколько серьёзна проблема холодного старта
- Выбор подхода зависит от ситуации — нет универсального ответа
- Всегда планируй решение холодного старта заранее
- Метрики определяй до запуска; финальная оценка — только через A/B тест
- Начинай просто, итерируй на основе данных

## Ключевые термины урока

- **Формализация задачи** — перевод бизнес-запроса в ML-постановку с чёткими метриками.
- **Выбор подхода** — решение между CF, контентным и гибридным методом на основе данных.
- **Холодный старт** — проблема, которую нужно учесть при проектировании системы.
- **Метрика успеха** — число, по которому решается: стало ли лучше после изменения системы.
- **Итерация** — запустить → оценить → улучшить → повторить.
